= Введение

Для хранения и организации большого количества данных в современных компьютерных системах используются системы управления базами данных (СУБД). Большинство соввременных промышленных решений (PostgreSQL, MySQL, Oracle) исполняют роль императивного интерфейса для доступа к данным. В рамках этого подхода внешние системы (клиенты) отправляют в СУБД точечные команды по обновлению (`UPDATE`, `INSERT`) и чтению данных.

Однако в данных системах СУБД обладает лишь поверхностной информацией о семантической взаимосвязи данных. Это приводит к ряду фундаментальных проблем:
1. *Проблема обеспечения целостности.* Контроль за соблюдением комплексных инвариантов делегируется внешним приложениям. СУБД, являясь пассивным хранилищем,  может гарантировать сохранность произвольных инвариантов.
2. *Сложность верификации.* Простые обновления данных могут требовать исполнений длинно пследовательности тяжёлых инструкций, корректность которых трудно доказать формально.
3. *Риск потери данных.* Неверная последовательность императивных команд может привести к необратимому повреждению состояния.
4. *Проблемы распредления.* Императивная архитектура затрудняет синхронизацию состояния между реплики, вынуждая многие системы опираться на централизованный узел для выполнения всех записей.

В рамках данной работы в качестве альтернативы императивной архитектуре предлагается *функционально-реактивная архитектура*. Ключевые особенности предлагаемой модели:
- Архитектура не предусматривает прямых операций перезаписи или удаления существующх данных. Взаимодействие с СУБД ограничивается добавлением записей в журнал событий (append-only).
- СУБД является *Event-Sourced*: упорядоченный список событий выступает единственным источником истины.
- СУБД автоматически выполняет организацию данных на основе заранее заданных разработчиком правил, преобразуя список событий в рамках многоступенчатого процесса в итоговое представление, доступное клиентам для запроса.

= Теоретические основы и архитектура

== Формальная модель журнала событий

Тривиальная модель, представляющих историю как линейный список событий $L = [e_1, e_2, dots, e_n]$, несостоятельна в распределённой среде из-за отсутствия глобального синхронизированного времени и необходимости слияния данных с независимых узлов.

Для построения устойчивой модели введём понятие *идентификатора события* $"id" in cal(I)$, исполняющего роль певичного ключа события. Множество $cal(Ш)$ должно быть *линейно упорядоченным*. В качестве реализации $cal(I)$ возможно рассмотреть использование множества лексикографически упорядоченных кортежей:
$ id = chevron.l t, n, c chevron.r $
где:
- $t$ — физическое время (Unix timestamp) регистрации события устройства-регистранта;
- $n$ — уникальный идентификтор узла-источника;
- $c$ — монотонно возрастающий счётчик узла-источника, разрешающий коллизии в рамках одной временой кваны.

Порядок задаётся следующим образом:
$
  id_A < id_B <=> & (t_A < t_B) \
                   & or (t_A = t_B and c_A < c_B) \
                   & or (t_A = t_B and n_A = n_B and c_A < c_B)
$

Тогда *журнал событий* $cal(L)$ определяется как конечно частичное отображение из множества идентификаторов $cal(I)$ в множество полезных нагрузок события $E$:
$ cal(L) : cal(I) harpoon E $

Подобное определение обладает двумя фундаментальными преимуществами:
1. *Простота объединения.* Объединение истории с двух экземпляров СУБД $cal(L)_A$ и $cal(L)_B$ сводится к теоретико-множественному объединению их графиков:
  $ cal(L)_("merged") = cal(L)_A union cal(L)_B$
2. *Согласованность.* Несмотря на то, что события могут поступать в систему в произвольном порядке, результирующий журнал является детерминированным, и его записи обладают строгим линейным порядком.

Физической реализацией данной модели журнала событий може являться двоичное дерево поиска, использующее идентификаторы $cal(I)$ в качестве ключей и хранящее полезную нагрузку $E$ в качестве значений.

== Формальная модель правил трансформации: централизованный подход

В основе предлагаемой системы лежит отказ от хранения текущего состояния как первичной сущности. Вместо этого состояния базы $cal(S)$ в определённый момент времени определяется функцией от истории событий $cal(L)$.

Наивный подход предполагает прямое задание функции $f : cal(L) -> cal(S)$. Однако, по мере роста числа событий в системе, полное восстановление состояния становится вычислительно невозможным за приемлемое время.

Для решения поблемы производительности перейдём к моделе *инкрементальных вычислений*. Определим *правило* образом, который допускает обработку не всего исходного объекта в рамках каждого вызова, а лишь его "приращения" по сравнению с предыдущим запросом. Например, в случае, когда исходным объектом является журналом событий, приращением можно считать лишь список новых появившихся в системе событий. Подобная инкрементальная обработка поступающих данных позволяет переиспользовать предыдущие результаты вычислений.

Определим множество внутренних состояний правила как $cal(C)$ ("cache"). Тогда инкрементальное правило в первом приближении можно представить как функцию:
$ "rule"_("top") : cal(C) times cal(L) -> cal(C) times cal(S) $

(Предъявляемые к внутреннему состоянию $cal(C)$ критерии корректности будут более подробно рассмотрены в секции @cache-irrelevance).

Данное представление является монолитным и централизованным: в нём всё состояние СУБД определяется единственным высокоуровневым правилом. Подобный подход формально корректен и находит применение в ряде моделей функционально-реактивного программирования (например, как Monadic Stream Functions @perez2016frp). Однако в контексте СУБД он приводит к избыточно связанности  логики и подходу "сверху-вниз", при котором затруднительно выделить последовательные этапы обработки.

== Формальная модель правил трансформации: композиции подход

В данной работе предлагается *композиционный подход*, основанный на декомпозиции общей логики трансформации на набор независимых автоматов с инкапсулированным состоянием. Подобная модель превращает процесс материализации данных в прозрачный многоступенчатый конвейер.

Ключевым требованием к модели является возможность комбинировать правила, обладающие различной структурой внутреннего состояния, в единый вычислительный граф. Для этого необходимо скрыть конкретный тип $cal(C)$ из внешней сигнатуры правила, оставив доступным лишь интерфейс для запроса материализованного значения.

Определим тип $R(K, O)$, параметризованый типом конфигурации $K$ и типом вывода $O$:
$ R(K, O) := exists cal(C) . (cal(C) times (cal(C) -> cal(M)_K (cal(C) times O))) $

Произвольный элемент типа $R(K, O)$ описывает одно правило системы и содержит:
1. Скрытый тип внутреннего состояния $cal(C)$, недоступный для внешних потребителей.
2. Начальное состояние $c_0 in cal(C)$.
3. Функцию перехода ("step"), оперирующую в контексте монады $cal(M)_K$. Она принимает текущее состояние $c in cal(C)$ и возвращает обновлённое состояние в комбинации с результатом вычисления.

Обратим внимание, что функция перехода не принимает журнал событий явно. Вместо этого доступ к журналу событий и результирующим объектам иных правил осуществляется через монадический интерфейс $cal(M)_K$. Монада $cal(M)_K$ выступает в роли операционной среды с параметрами типа $K$, ассоциированный со следующими примитивами:
- *Материализация результатов иных правил:*
  $"resolve" : R(K, O') -> cal(M)_K(O')$ — материализует значение иного правила. Позволяет правилу задействовать результаты вычисления иных правил.
- *Чтение журнала события:*
  $"events" : cal(M)_K(cal(L))$ — возвращает срез журнала событий $cal(L)$ на момент запуска операционной среды.
- *Чтение текущих конфигурационных параметров:*
  $"ask" : cal(M)_K(K)$ — возвращает текущие параметры конфигурации типа $K$. Подобный механизм позволяет динамически конфигурировать поведение существующего правила без необходимости сбрасывать его внутреннее состояние.
- *Локальное обновление конфигурации:*
  $"local" : (K -> K') -> cal(M)_(K')(cal(O)) -> cal(M)_K(cal(O))$ — вызывает монадическое подвычисление с обновлённой конфигурацией. Это позволяет высокоуровневым правилам осуществлять ограниченный контроль над поведением низкоуровневых правил.
Формально монада $cal(M)_K$ может быть представлена в виде стека монадических трансформеров, объединяющего контекст чтения конфигурации (монада `Reader`), контекст управления реестром внутренних состояний правил (монада `State`) и контекст доступа к актуальному срезу журнала событий (монада `Reader`).

=== Ограничение монадической модели

Реализация операционной среды с использованием монады $cal(M)$ предоставляет максимальную выразительность (с возможностью динамического выбора дальнейшего поведения системы на основе предыдущих шагов), но накладывает ряд ограничений на эффективность системы управления памятью и сборку мусора.

Динамическая природа монадической композиции вызывает следующие трудности:
1. *Непрозрачность графа вычислений.*  Невозможность построения статического графа зависимостей затрудняет предварительную оптимизацию запросов и возможность параллельного исполнения независимых правил.
2. *Сложность управления жизненным циклом кэша.* В динамической модели затруднительно определить момент, когда внутреннее состояние конкретного правила становится избыточным и может быть безопасно удалено из памяти для сохранения физической памяти.

В качестве перспективного решения рассматривается полный или частичный переход к *аппликативным функторам* или *селективным функторам* (Selective Applicative Functors @mokhov2019selective) для статического описания структуры зависимости.

Данный подход позволяет описать граф преобразований декларативно, до начала обработки данных, что открывает возможности для:
- Статического анализа потребления памяти и детерминированной сборки мусора.
- Автоматического распараллеливания вычислений.

В рамках текущего исследования используется монадическая модель как наиболее гибкая и простая для этапа прототипирования, однако вопрос перехода к строгому аппликативному интерфейсу остаётся открытой задачей для дальнейшего исследования.

== Требование незначимости кэша и детерминизм вычислений <cache-irrelevance>

Введение внутреннего состояния правила ($cal(C)$) и использование монадического контекста $cal(M)_K$ позволяет описывать нетривиальные инкрементальные вычисления, но создаёт риск нарушения функциональной чистоты системы. Если результат вычисления правила зависит от содержимого кэша, СУБД потеряет свойство детерминизма, и материализуемое состояние перестанет быть однозначным.

Например, если система чувствительна к порядку поступления событий в журнал, то синхронизация журналов состояний нескольких независимых экземпляров СУБД может привести к различному итоговому состоянию. Для решения этой проблемы вводится требование *незначимости кэша*.

Пусть $Gamma$ — множество реестров состояний всех правил системы. Обозначим процесс материализации правила $R(K, S)$ в конфигурации $K$ на основе журнала событий $cal(L)$ с начальным реестом $Gamma$ как функцию:
$ "eval" : R(K, S) times cal(L) times K times Gamma -> S times Gamma $

Будем говорить, что правило $r in R(K, S)$ удовлетворяет требованию незначимости кэша, если для любого журнала $l in cal(L)$, конфигурации $k in K$ и любых двух произвольных состояний реестра $gamma_1, gamma_2 in Gamma$ выполняется тождество:
$ "fst" ("eval" (r, l, k, gamma_1)) = "fst" ("eval" (r, l, k, gamma_2)) $
где $"fst"$ — проекция, извлекающая первый элемент прямой декартового произведения $S times Gamma$.

Соблюдение требования незначимости кэша является критически важным для обеспечения целостности материализованного состояния, однако его проверка в языках общего назначения затруднена. В рамках данной работы предлагается использовать аппарат формальной верификации @chlipala2013certified, основанный на соответствии Карри — Ховарда.

При этом подходе, требование незначимости кэша может быть представлено как тип данных, а доказательство соответствия правила требованию выражается в виде экземпляра данного типа. Таким образом, в языке с поддержкой формальной верификации разработчик способен не просто описать инкрементальную функцию трансформации $r in R(K, O)$, но и статически гарантировать её эквивалентность некой "эталонной" чистой функции $f : cal(L) times K -> O$.

Соответствие Карри — Ховарда также позволяет гарантировать требуемые инварианты путём наложения ограничений на конфигурацию $K$, внутреннее состояния конкретного правила $cal(C)$, материализуемый тип $O$. Возможность переложить ответственность за проверку инвариантов с человека на компилятор является веским аргументом в пользу спользования специализированного языка для описания внутренних правил СУБД.

= Реализация прототипа и эволюция вычислительной модели

Для верификации предложенной архитектуры СУБД был разработан прототип на языке Haskell. Выбор данного инструмента обусловлен развитой системой типов, поддержкой алгебраических типов данных и чистотой функций, что идеально соответствует концепции функционально-реактивной обработки данных. Данная глава посвящена процессу разработки прототипа и критическому анализу принятых архитектурных решений.

== Haskell как язык задания правил и агностицизм модели данных

Ключевым архитектурным решением, определившим облик прототипа, стал отказ от использования промежуточного языка правил на раннем этапе. Вместо этого в качестве языка для задания правил и осуществления трансформаций решено использовать обычные функции языка реализации (Haskell).

Данное решение позволило в рамках прототипа:
- Задействовать всю мощь языка общего назначения для задания правил.
- Обеспечение агностицизма к моделе данных. Большинство промышленных СУБД навязывают жёсткий формат организации данных (например, реляционный, документоориентированный, графовый), что зачастую порождает несоответствие между структурами данных в памяти приложения-клиента и способом их хранения в базе данных. В разработанном прототипе СУБД является агностиком по отношению к конкретному формату данных: система оперирует не кортежами или документами, а произвольными сериализуемыми объектами (например, деревья).

Таким образом, схема базы данных определяется исключительно системой типов используемого языка правил.

Недостатком данного решения оказалось необходимость вырабатывать в рамках прототипа механизмы для сериализации произвольных объектов. Наибольшую трудность представили замыкания, которые не могут быть сериализованы встроенными средствами Haskell.

= Архитектура ресурсов

Прямым следствием отказа от фиксированного формата данных стала необходимость реализации комплексной подсистемы управления памятью. Поскольку СУБД оперирует произвольными индуктивными типами данных (деревьями, списками, графами), возникает проблема эффективной десериализации крупных объектов.


Рассмотрим задачу поиска одного объекта в большой таблице. Классическая реализация реляционных СУБД зачастую требует лишь частичной загрузки таблицы в оперативную память СУБД. С другой же стороны, в контексте прототипа "таблица" реализуется в качестве единственного объекта Haskell. Полная загрузка и десериализация этого объекта в оперативную память для выполнения точечной операции не является приемлемой в условиях ограниченных ресурсов.

Для организации данных, находящихся на разных уровнях иерархии памяти, в рамках прототипа была введена абстракция `Res`. `Res` действует как умный указатель, абстрагирующий физическое положение данных и обеспечивающий загрузку и десериализацию тяжёлых структур данных по мере необходимости.
$ "Res"(A) := "ResInline"(A) | "ResBuiltin"("Word64", A) | "ResAlloc" ("ResA"(A)) $
где:
- `ResInline` — прямое хранение ресурса. Применимо для простейших ресурсов, не нуждающихся в отдельных аллокациях.
- `ResBuiltin` — встроенный в прототип ресурс, обладающий статическим идентификатором (64-битным числом).
- `ResAlloc` — динамический ресурс, который потенциально требует десериализации при первом обращении.

Широко известным приёмом функциональных языков программирования является применение ленивых вычислений и мемоизация. В случае Haskell, модель ленивых вычислений позволяет конструировать "бесконечные" структуры данных, генерируемые по мере необходимости. Вычисленные подчасти "бесконечной" структуры при этом сохраняются и не вычисляются заново.

Классическим примером подобной "бесконечной" структуры данных является реализация списка последовательности Фибоначчи на Haskell, представленная в листинге @fib.
#figure(
  ```hs
  fibs :: [Integer]
  fibs = 0 : 1 : zipWith (+) fibs (tail fibs)
  ```,
  caption: [Реализация "бесконечного" списка последовательности Фибоначчи на Haskell]
) <fib>

Для достижения подобного эффекта в рамках СУБД был реализован умный указатель `Delay`, являющийся расширением `Res` и допускающий хранение отложенных вычислений и кэширование их результатов. В то время как `Delay` позволил достичь желаемого результата, временные задержки и непредсказуемое использование памяти, связанные с ленивыми вычислениями, делают использование `Delay` в реальных сценариях непрактичным и опасным.

== От мемоизации к инкрементальным автоматам

Первоначальная концепция трансформации данных опиралась на математически элегантную, но практически несостоятельную модель полной проекции истории:
$ f : cal(L) -> S $

В рамках данной модели предполагалось, что ленивая природа `Delay` и реализация мемоизации позволит системе автоматически кэшировать промежуточные результаты вычислений. Например, если журнал событий $cal(L)$ увеличивается на одно событие $e$, то ленивое дерево вычислений должно обновить только те узлы, которые напрямую зависят от $e$, переиспользуя старые вычисления для иных частей дерева благодаря мемоизации.

Более подробный анализ выявил, что:
- Полная мемоизация является крайне затратной по памяти;
- Не все трансформации тривиально представимы через рекурсию;
- Рекурсивный подход к описанию трансформации усложняет обработку примитивных обновлений.

Для преодоления этих ограничений был совершён переход к более гибкой модели явного управления состоянием, реализованной через абстракцию `Gear` ("шестерня").

Концепция Gear переводит правила из категории "функция от истории" в категорию "автомат Мура". В этой модели правило обладает явным типом внутреннего состояния $cal(C)$ и функцией перехода.

Однако при реализации этой модели возникла фундаментальная проблема динамической конфигурации. В реальных системах возникает необходимость смены поведения системы в ходе её работы, а любое изменение кода функции требует полного сброса внутреннего состояния и пересчёта всей истории.

Для решения этой задачи была разработана структура, поддерживающая механизм подмены конфигурации без потери внутреннего состояния. Итоговая реализация `Gear` в рамках прототипа представлена в листинге @gear-hs.

#figure(
  ```hs
  -- | Шаблон Gear, не имеющий внутренней конфигурации и состояния. Может быть преобразован в GearFn путём конфигурации.
  data GearTemplate ctx (out ∷ Type) cache cfg = UnsafeGearTemplate !cache !((ctx, W1 Maybe cfg) :-> M AppIOC cfg) !((cfg, cache) :-> M AppIOC (out, cache))
    deriving (Generic)

  -- | GearFn, сконфигурированная версия GearTemplate, имеющая внутреннюю конфигурацию, о не состояние.
  data GearFn ctx out cache = ∀ cfg. GearFn !(ValT cfg) !cfg !(GearTemplate ctx out cache cfg)

  -- | Gear, комбинация GearFn с внутренним состоянием.
  data Gear ctx out = ∀ cache. UnsafeGear !(ValT cache) !(GearFn ctx out cache) !Int
  ```,
  caption: [Реализация абстракции `Gear` в рамках прототипа]
) <gear-hs>


`GearTemplate` состоит из трёх компонентов:
1. *Начальное внутреннее состояние `cache`.*
2. *Функция инициализации/миграции конфигурации:* Функция `(ctx, W1 Maybe cfg) -> M AppIOC cfg` принимает текущий контекст и, потенциально, предыдущую конфигурацию, позволяя вычислить новую конфигурацию `cfg`.
3. *Функция шага:* Функция `(cfg, cache) -> M AppIOC (out, cache)`, выполняющая обновление внутреннего хранилища и производящая объект типа `out`.

Подобная архитектура позволила достичь высокой вычислительной эффективности: при постулпении новых событий система не пересчитывает всё состояние, а лишь вызывает функцию шага для затронутых правил. Задействование экзистенциальных типов (`forall cfg cache`) позволило создать дерево правил, в котором каждое правило инкапсулирует свою внутреннюю конфигурацию и состояние.

Заметим, что подобная реализация реактивных ячеек допускает задание правил с использованием рекурсии и мемоизации, так что данная модель напрямую является более мощной версией исходной концепции $f : cal(L) -> S$.

== Композиция Gear

Несмотря на гибкость модели `GearTemplate`, ручное описание всех трёх компонентов (начального состояния, функции миграции и функции шага) для каждого правила сопряжено с написанием большого количества однотипного ("boilerplate") кода. Особенно остро эта проблема проявляется при попытке объединить несколько простых правил в одно сложное: разработчику приходится вручную описывать типы-произведения для объединения кэшей ($cal(C)_1 times cal(C)_2$) и конфигураций ($K_1 times K_2$).

Для решения этой задачи в рамках прототипа была разработана аппликативная структура `Asm` ("Assembly"), которая берет на себя управление жизненным циклом иерархических состояний.

С точки зрения теории категорий, `Asm` является аппликативным функтором, для которого определены операции `pure` и `<*>`. Это позволяет комбинировать правила декларативно:

$ "pure" : O -> "Asm"("Ctx", O) $
$ "<*>" : "Asm"("Ctx", O_1 -> O_2) times "Asm"("Ctx", O_1) -> "Asm"("Ctx", O_2) $

При такой декомпозиции движок СУБД автоматически выстраивает дерево состояний и конфигурации, соответствующее дереву композиции `Asm`. Это гарантирует, что каждое под-правило получит именно ту часть общего состояния и конфигурации, которая ему принадлежит, обеспечивая строгую изоляцию и типобезопасность.

Одним из наиболее выразительных примеров использования этой абстракции является функция буферизации `buffered`. Она позволяет любому правилу "запоминать" свой предыдущий результат, превращая его из мгновенного значения в пару "предыдущее-текущее":

$ "buffered" : O_("init") -> "Asm"("Ctx", O) -> "Asm"("Ctx", O times O) $

Реализация `buffered` (Листинг @buffered-hs) внутри использует примитив `asmCached`, который выделяет дополнительную ячейку памяти в иерархии внутреннего состояния для хранения значения с предыдущей итерации.

#figure(
  ```hs
  buffered :: (InferValT True out) => out -> Asm ctx out -> Asm ctx (out, out)
  buffered cache0 outA = asmCached cache0 do
    out <- outA
    pure $ \old -> pure ((old, out), out)
  ```,
  caption: [Реализация комбинатора буферизации на основе Asm]
) <buffered-hs>

Данный комбинатор оказывается крайне полезным для реализации системы отслеживания изменений между старой и новой версией, на которой построены многие инкрементальные правила.

== StateGraph и ретроактивные изменения

Одной из фундаментальных сложностей в Event-Sourced системах является корректная обработка ретроактивных изменений и поддержание согласованности данных вне зависимости от порядка поступления событий.

Пусть правило $r_a$ при вычислении состояния объекта $x$ в момент времени $t_2$ обращается к состоянию объекта $y$, вычисленному правилом $r_b$ в момент $t_1$. Если в систему поступает новое событие $e$, изменяющее состояние $y$ в момент $t_1$, то система должна инициировать пересчёт состяния $X$.

Для решения этой задачи в рамках прототипа был разработан механизм `StateGraph`. Его основной задачей является отслеживание графа зависимостей между состояниями различных объектов и автоматическое выполнение пересчёта для состояний всех объектов, косвенно затронутых текущим обновлением. Реализация `StateGraph` в прототипе хранит хронологию каждого объекта, включающую в себя все его изменения состояния и моменты, когда состояние объекта было считано для формирования состояния иного объекта.

Алгоритм распространения обновлений реализван через очередь пересчёта. При изменении состояния в узле графа, все связанные с ним "события-слушатели" помещаются в общую очередь и пересчитываются. Для хранения хронологии используется сжатое префиксное дерево, что позволяет эффективно выделять лишь тот диапазон записей, что следует после момента обновления.

== Тестирования прототипа

Для итоговой проверки корректности реализованных механизмов (`Res`, `Gear`, `Asm`, `StateGraph`) был разработан тестовый сценарий, моделирующий систему управления доступа RBAC на основе событий.

=== Описание модели предметной области

В качестве тестовой модели выбрана система управления доступа пользователей. В листинге @model-hs представлены перечень ролей `SiteAccessLevel` и набор событий `Event`.

#figure(
  ```hs
  data SiteAccessLevel = SalNone | SalUser | SalModerator | SalAdmin
    deriving (Eq, Show, Generic)

  data Event
    = CreateUser -- Регистрация нового пользователя
    | AdminSetAccessLevel !(W1 Maybe UserId) !UserId !(W SiteAccessLevel)
  ```,
  caption: [Фрагмент модели данных тестового сценария]
) <model-hs>

С задействованием `StateGraph` реализовано правило `status`, отслеживающее и обновляющее состояние всех пользователей. Исходный код правила представлен в листинге @status-hs. При поступлении в систему события изменение уровня доступа `AdminSetAccessLevel` выполняется проверка, что инициатор события (`adminM`) отсутствует (что означает, что событие отправлено суперпользователем), или существует и сам обладает правами администратора (`SalAdmin`). Считывание текущего состояния инициатора выполняется с использованием команды `SG.query`, а фиксация результата выполняется через `SG.update`.

#figure(
  ```hs
  status = SG.mkStateGraph (asmGear events) (SG.StateGraphDepsNil, \case
    AdminSetAccessLevel (W1 adminM) target level -> do
      hasAccess <- case adminM of
        Nothing -> pure True -- Суперпользователь
        Just admin -> SG.query admin <&> \case
          Just (W SalAdmin) -> True
          _ -> False
      when hasAccess $ SG.update target $ pure level
    _ -> pure ()
  )
  ```,
  caption: [Реализация правила проверки прав доступа через StateGraph]
) <status-hs>

=== Методика и результаты верификации

Для подтверждения корректности механизмов ретроактивного обновления был разработан тест на основе библиотеки `QuickCheck`. Основная задача тестирования заключается в проверке того, что материализованное состояние системы инвариантно относительно порядка поступления событий при сохранении меток `EventId`.

В листинге @test-logic представлены ключевые функции тестирования:
- `oneshot` (обработка журнала событий целиком);
- `multishot` (инкрементальная обработка случайных подмножеств журнала).

#figure(
  ```hs
  oneshot ∷ [(EventId, W Event)] → (SG.StateGraph EventId (W SiteAccessLevel) → AppIOC r) → r -- [(UserId, [(EventId, SiteAccessLevel)])]
  oneshot t renderer = unsafeRunAppIO do
    status' ← confNewGear status ()
    putEventList t
    renderer =<< runGear status'

  prop_test1_oneshot_correct =
    withMaxSuccess 1
      $ test1Res
      == oneshot test1 SG.toLists

  multishot ∷ [(EventId, W Event)] → (SG.StateGraph UserId (W SiteAccessLevel) → AppIOC a) → a
  multishot t renderer = unsafeRunAppIO do
    status' ← confNewGear status ()
    for_ @[] (inits t) \curr → do
      _ ← runGear status'
      putEventList curr
    renderer =<< runGear status'

  {- | This test shuffles the list of events and provides events to Dentrado one by one.
  It is expected that any shuffle of the input events yields the same result, since
  all events are associated with some point in time.
  Dentrado, being reactive, processes these events incrementally, but might
  perform expensive history rewrites to keep the result consistent.
  -}
  prop_test1_multishot_correct = withMaxSuccess 100 $ forAllShrink
    (shuffle test1)
    (shrinkList shrinkNothing)
    \test1' →
      oneshot test1' SG.toLists == multishot test1' SG.toLists

== Анализ результатов прототипирования и постановка задачи

Реализация прототиа на Haskell подтвердила теоретическую жизнеспособность предложенной архитектуры. Были успешно реализован механизм инкрементальных вычислений (`Gear`), композиции (`Asm`), ретроактивных изменений и поддержания согласованности (`StateGraph`). Однако в ходе разработки были выявлены ограничения, связанные с использованием языка общего назначения для описания логики СУБД:

1. *Проблема сериализации функций.* В то время как функциональная парадигма допускает использование функций высшего порядка, их практическое применение в рамках прототипа оказалось затруднено по причине невозможности сериализации замыканий Haskell встроенными механизмами. Для частичного обхода этой проблемы пришлось задействовать статические реестры функции, тяжёлых в обслуживании. Желательным является использование для СУБД языка, в котором произвольные функции являются сериализуемыми и могут быть напрямую сохранены на диске или переданы по сети.

2. *Отсутствие инструментария фомальной верификации.* В то время как система типов Haskell позволяет выражать сложные инварианты, она не предназначена для полноценного доказательства теорем. Для реализации требования "незначимости кэша" (описанный в секции @cache-irrelevance) необходимо иметь возможность на уровне типов доказывать эквивалентность инкрементального алгоритма и его чистой спецификации. Отсутствие поддержки формальное верификации в языке оставляет риск рассинхронизации материализованного состояния и исходного журнала событий.

3. *Отсутствие гарантий тотальности.* Haskell является полным по Тьюрингу языком, что допускает ошибочное описание правил с незавершаемыми вычислениями. Таким образом, неправильно сформулированное правило в контексте СУБД может привести к отказу всей системы. Желательно, чтобы используемый язык позволял гарантировать завершимость (тотальность) вычислений.

4. *Проблема циклических структур данных.* Haskell позволяет конструировать циклические структуры данных. Однако наличие циклов в графе данных значительно усложняет организацию дискового пространства и механизм сборкимусоа. Для эффективной работы целесообразно ограничить СУБД правилами, порождающими исключительно направленные ациклические графы.

5. *Переусложнённость.* Haskell является языком общего назначения, представляющм иснтрументы для решения шиокого спектра задач. Для специфической задачи описания трансформаций данных такая универсальность является избыточность.

На основе данных выводов формулируется задача разработки специализированного функционального языка программирования для описания правил СУБД. Ключевые требования к языку:
- *Минималистичность и простота базового освоения.*
- *Полная сериализуемость всех сущностей*, включая замыкания.
- *Зависимая типизация и функционал формальной верификации* для поддержки доказательств корректности трансформаций.
- *Гарантия тотальности* для исключения бесконечнх циклов на этапе проверки типов.
- *Ацикличность данных:* запрет на создание циклических ссылок на уровне семантики языка.

Реализация компилятора данного языка рассматривается в следующей секции.

= Разработка языка Fadeno

Для преодоления выявленных ограничений был разработан специализированный язык Fadeno. Это функциональный декларативный язык с градуированной, внешней зависимой типизацией.

Язык спроектирован так, чтобы код правил можно было хранить, передвать по сети и исполнять в изолированной среде СУБД, при этом предоставляя математические гарантии корректности выполняемых операций.

Пример кода на Fadeno с общей демонстрацией синтаксических конструкций представлен в приложении @fad-example.

== Архитектурные принципы и семантика

Fadeno вдохновлён языками Dhall и Cedille @stump2017cdle. В его основе лежат следующие принципы:
1. *Всё есть выражение.* В языке отсутствует разделение на инструкции и выражения. Любая конструкция — от литерала числа до объявления лямбда-функции — является выражением, возвращающм значение. Это упрощает парсинг, увеличивает модульность и упрощает композицию.
2. *Внешняя (extrinsic) типизация и стирание, градуированная типизация.* Типы в Fadeno и определённые конструкции существуют исключительно для проверки корректности типов и стираются на дальнейших этапах компиляции, не влияя на них. Этот подход закрепляет роль статической типизации исключительно как гаранта безопасности. Градуированная модель типизации позволяет языку функционировать даже если программа не проходит полностью проверку типов. Это упрощает быстрое прототипирование, делает процесс компиляции предсказуемым и упрощает первоначальное освоение языка.
3. *Сериализуемость.* Все объекты в Fadeno, включая лямбда-функции, сериализуемы.
4. *Тотальность и поддержка формальной верификации.* Полностью типизированные программы на языке тотальны (гарантированно завершают работу за конечное время). Язык поддерживает функционал для доказательства теорем и функциональной верификации.

== Зависимые типы

Ядром Fadeno является поддержка зависимых типов, где типы могут зависеть от значений. В контексте СУБД это позволяет описывать комплексные структуры данных, в которых типизация одной части структуры зависит от значения иной. Примером тако структуры является tagged union:
Tagged union, или "дискриминирующее объединение", содержит в себе численный тег, описывающий вариант, и значение, соответствующее этому варианту. Дискриминирующее объединение является идеальной демонстрацией зависимых типов, так как тип хранимого в нём  значения зависит от значения численного тега.

== Типизация

При номинативной типизации два типа считаются эквивалентными только если они порождены в рамках одного и того же определения. При структурной типизации два типа совместимы, если их внутренняя структура совпадает, вне зависимости от происхождения типа.

Fadeno использует по умолчанию структурную типизацию, допуская использование номинативной типизации для инкапсуляции внутреннего устройства типов. Кроме того, Fadeno поддерживает понятие подтипов, что позволяет напрямую использовать объекты более узкого подтипа в более широком контексте без необходимости в ручной конвертации (например, использовать натуральное число в контексте, ожидающим целое число).

Проверка выражений на соответствие типу выполняется в двунаправленном режиме, при котором компилятор автоматически проверяет выражение на соответствие данной им аннотации, а также автоматически синтезирует тип для неаннотированных выражений.

Эти решения значительно снижают количество однотипного кода и упрощают написание логических корректных приложений.

== Записи

Для СУБД критически важна работа со схемами данных. Для хранения документных данных Fadeno применяет записи. Записи являются набором полей, где каждое поле обозначено тегом (уникальные идентификатором, начинающимся с точки и имеющим вид `.custom_tag`) и может иметь отличный от иных тип. "Строка" (row) является перечислением списка полей соответстующей записи на уровне типов.

К значению поля `.field` поля `record` можно обратиться, задействуя синтаксис `recodrd.field`.

Язык реализует *зависимые записи* и *полиморфизм строк* (row polymorphism) через экспериментальную конструкцию зависимой конкатенации строк (`\./`). Зависимая конкатенация позволяет конструировать сложные строки, описывающие зависимые записи, как комбинацию простых строк.

Пример использование зависимой конкатенации для построения зависимой записи представлен в листинге @option. `Option(V)` выражается как зависимая конкатенация строки с единственным полем `.has_value` типа `Bool` со строкой, структура которой определяется хранимым в поле `.has_value` значении.

#figure(
  ```
  Option = \V. {( .has_value = Bool )} \./ (if ..has_value { .v = V } {})
  ```,
  caption: [Реализация Option-ячейки с использованием зависимой записи]
) <option>

Зависимая конкатенация строк является вариацией использования зависимого пересечения @intersection для определения зависимых записей. Особенностью конкатенации строк является предсказуемое поведение, возможность наслоения нескольких полей с одинаковым именем друг на друга и упрощённая логика проверки типов на совместимость.

== Тотальность

Для использования в ядре СУБД критично, чтобы пользовательские правила гарантированно завершались (отсутствие бесконечных циклов). Fadeno является *тотальным* языком.

Вместо неконтролируемой рекурсии (Y-комбинатора), язык предоставляет встроенный примитив `loop`, реализующий принцип структурной рекурсии с метрикой. Его сигнатура:

$"loop" : "Fun" \{I\} \{"measure" : I -> "Int"+\} \{O\} \ (I) ("Fun" ("curr" : I) ("Fun" ("next" : I) \{_ : "measure" "next" < "measure" "curr"\} -> O) -> O) \ -> O$

Это означает, что для организации цикла разработчик обязан предоставить:
1.  Функцию-метрику, отображающую состояние цикла в неотрицательное целое число (`Int+`).
2.  Доказательство того, что на каждой итерации метрика строго убывает. Поскольку последовательность неотрицательных целых чисел не может убывать бесконечно, это статически гарантирует завершение программы.

Для избежания парадокса Жирара, возникающего при допущении $"Type" : "Type"$, Fadeno реализует иерархию вселенных типов: вместо единственного типа `Type` язык предоставляет встроенную функцию `Type^`, возвращающую вселенную типов для каждого натурального числа:
$ "Type" eq "Type^" 0 : "Type^" 1 : "Type^" 2 : dots $

== Механизм формальной верификации и Rewrite Rules

Fadeno использует соответствие Карри — Ховарда для доказательства теорем. Тип `Eq a b` представляет собой доказательство равенства выражений `a` и `b`.

Уникальной особенностью компилятора является поддержка пользовательских правил перезаписи (`rewrite`). Если в контексте существует доказательство `proof : Eq a b`, конструкция `rewrite proof in expr` инструктирует компилятору автоматически подставлять выражение `b` вместо каждого встреченного выражения `a` внутри выражения `expr`. Это — крайне мощный инструмент, позволяющий просто описывать тяжёлые доказательства.

Для дальнейшего упрощения рутинных доказательств Fadeno поддерживает ряд арифметических упрощений на этапе нормализации, задействуя сведение произвольных алгебраических выражений к полиномиальной нормальной форме.

// Комментарии

// Распаковывеет Int, Type^, List, if... из встроенной библиотеки "fadeno" в едущий контекст
unwrap fadeno. Int Type^ List if Eq refl

// Задание констант
x = true

// Помещение анонимной функции в константу
add =
  // Объявление анонимной функции от аргументов "a" и "b"
  \a b.
    // Вызов встроенной функции fadeno.int_add с сохранением результата в константу result
    result = fadeno.int_add a b
    in result

// Константам можно явно задавать тип
/: Int
y = 4

// Анонимные функции имеют тип `Fun (A) (B) (C) -> Z`
/: Fun (Int) (Int) -> Int
constf = \arg1 arg2. arg2 // игнорирует первый аргумент, возвращает второй.

// Списки
/: List Int
list = [1 | 3 | 5 | 4 | -1]

// Записи
/: {( .field1 = Int | .field2 = Fun (Int) (Int) -> Int | .zer = List Int )}
record =
  { .field1 = 4
  | .field2 = constf
  | .zer = [ 1 | 3 ]
  }

accessedField1 = record.field1

// Type Alias
/: Type^ 0
MyRecord = {( .a = List Int )}

/: MyRecord
my_record = { .a = [] }

// Type universes
/: Type^1
Type = Type^ 0
/: Type^ 2
Kind = Type^ 1

// Полиморфизм
/: Fun {A} (A) -> A
id = \@A x. x

// Доказательство теорем
/: Fun {a} {b} {c} (Eq a b) (Eq b c) -> Eq a c
proof = \ab bc.
  rewrite ab
  in bc

// Зависимые типы
/: Fun (x : Bool) -> if x (Int) (List (List Int))
cond = \x. if x
  (\@x_is_true. rewrite x_is_true in 3)
  (\@x_is_false. rewrite x_is_false in [[3]])

// Импорт
other_file = ./other_file

in {}
```
